# Лог гипотез (архитектура и модели)

Цель: каждое изменение, на которое вы рассчитываете как на **улучшение метрик**, имеет **идентификатор**, **формулировку**, **реализацию в коде** и **результат тестов**. Без этого нельзя отличить удачную идею от переобучения на шуме.

## Принципы

1. **Одна гипотеза — один фокус** (например: только веса агрегатора L5, или только порог T1 гейта, или только промпт). Иначе неясно, что сработало.
2. **`model_id` / `run_id`** в JSONL оценки должны ссылаться на строку лога (например `H-014`).
3. **Критерий приёмки** задаётся до прогона: какая метрика на каком датасете (например Spearman IC по `forward_log_return_1d` на фикстурах апреля, отдельно страта `event_tag=geo`).
4. **Статус**: proposed → implemented → tested → kept / reverted / deferred.

## Шаблон записи

Скопируйте блок в конец файла или в отдельный `hypothesis_log_entries.md`.

```text
### H-XXX — краткое название
- **Дата:** YYYY-MM-DD
- **Гипотеза:** Если мы изменим …, то метрика … улучшится, потому что …
- **Область:** gate | L5 prompt | агрегатор | каналы | cheap_sentiment | fusion | датасет
- **Реализация:** ветка / коммит / файлы (пути в nyse или tradenews)
- **model_id в оценке:** например `ollama:qwen2.5:7b|H-XXX`
- **Датасет / фикстуры:** `fixtures/articles/...`, список дат, фильтр event_tag
- **Метрики:** IC, hit rate, бакеты confidence, end-to-end (опционально)
- **Результат (после тестов):** числа + краткий вывод
- **Статус:** kept | reverted | deferred
- **Заметки:** что поменять в следующей итерации
```

## Связь с репозиторием

| Артефакт | Назначение |
|----------|------------|
| `docs/fixtures.md` | Откуда берутся фиксированные новости |
| `examples/*.jsonl` / прогоны `run_compare.py` | Численное сравнение `model_id` |
| Коммиты с префиксом `[H-XXX]` (по желанию) | Быстрый поиск в git |

Итог: **варьирование архитектурой и моделями** остаётся осмысленным, только если каждая ветка гипотезы имеет запись здесь и измеримый исход; иначе «более высокие метрики» на одном случайном срезе не воспроизводятся.
