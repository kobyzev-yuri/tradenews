# План: новостной импульс — внедрение и проверки

Документ фиксирует **намеченный порядок работ** и **типы тестов** с учётом уже пройденного бенча (мало точек, разные горизонты, Gemini/Qwen сильнее llama 3b на game-срезе, ProxyAPI, отдельность ценового импульса в GAME_5M). План пересматриваем после каждого заметного прогона или смены промпта/моделей.

---

## 1. Факты из опыта (не забывать)

- **Операционная реальность LSE (2026):** **`knowledge_base` новостей** в основном накапливается импортом/ботом, но **автоматическая игра сейчас на технике**; новости + LLM использовались **точечно человеком** (запросы в аналитику), часто как **ограничитель агрессии** (BUY / STRONG_BUY после разбора ленты). Значит схему KB и пайплайн наполнения **можно править под будущий сигнал** без страха «сломать уже работающий новостной автотрейд» — его как раз предстоит **включать осознанно** (§2, §7).
- **Корреляции:** в решениях LLM уже смешиваются **новости и техника** и учитывается **корреляция цен по тикерам** (портфель и контекст в промпте). Отдельно **не делалась** модель совместного появления/тематики новостей по тикерам («новостная корреляция») — при расширении KB и fusion это отдельный предмет гипотез.
- Оценка — **не обучение**: метрики по **тому JSONL точек**, который собрали; **тикерный состав** задаётся файлом датасета, а не отдельным «списком обучения» в коде бенча.
- **Малое N** (десятки точек) → смотреть **CI / p-value**, повторять на другом периоде; «победитель модели» может поменяться.
- **Новости ≠ вся цена**: техника и режим рынка могут перебивать новостной слой — в проде нужен **явный слой смешивания**, а не замена техники.
- **`GAME_5M_TAKE_MOMENTUM_FACTOR`** — про **ценовой** импульс 2ч; новости подключать **отдельными** правилами (фильтр, размер, опционально цель выхода), не смешивать смысл с этим коэффициентом.
- **Объём и качество потока новостей** — отдельное узкое место: мало статей, дубликаты, лимит токенов, задержка данных.

## 2. Внедрение (по этапам)

| Этап | Действие | Критерий готовности |
|------|----------|---------------------|
| **A** | При «решении» сохранять снимок сигнала: `ticker`, `decision_ts_utc`, `news_bias`, `confidence`, `model_id`, ссылки/id статей (полный текст — по возможности вне горячей БД). | Можно воспроизвести строку уровня EvaluationRow офлайн. |
| **B** | Слой **fusion** с техникой: использовать поле `tech_bias` в датасете / то же в проде; правило согласия знаков или взвешенный скор. | Снижение конфликтных входов без ухудшения метрик на holdout-срезе. |

| **D** | Регулярный мониторинг: сводка vs факт (хотя бы hit по знаку + разрез по тикерам); сравнение **Qwen vs Gemini** на новых точках. | Регресс сигнала виден за 1–2 недели. |

## 3. Дополнительные тесты (узкие места по новостям)

Все пункты — **офлайн** на том же или расширенном JSONL, без обязательной смены модели.

1. **Страты по числу статей** на точку (например 0–2 / 3–7 / 8+): Spearman и hit отдельно.  
2. **Усечение контекста:** заголовки only / топ-K по времени vs полный текст — проверка упора в **токены** и шум.  
3. **Дедуп** URL/заголовка vs сырой поток — влияние **повторов** на bias.  
4. **Лаг данных:** сдвиг `decision_ts` (симуляция поздней ленты) — устойчивость.  
5. **Новый временной срез** при тех же тикерах — не переобучиться на одном календарном окне.  
6. **Порог \|bias\|** (0.03 / 0.05 / 0.1): баланс покрытие vs hit rate.

## 6. Данные в LSE / NYSE: новости, свечи, схема «обучения»

Сейчас в LSE новости часто попадают в **`knowledge_base`** (`ts`, `ticker`, `source`, `content`, `sentiment_score`, `insight`) через `news_importer.py` и бота. Для **точного** импульса и последующей калибровки этого мало: нужны **дедуп**, **внешний id/url**, **часовой пояс**, **сырые свечи** с идемпотентным добором.

**Идея:** ничего из «игрового» ядра не удаляем; **`quotes`** и **`trade_history`** остаются; новый контур — **расширенная KB + бары market_* + лог новостного сигнала**. Детали по шагам — §6.0–6.5.

### 6.0 Как меняются основные таблицы в PostgreSQL (по этапам)

Ниже — **фактические имена** из **`lse/init_db.py`** и миграций **`lse/db/knowledge_pg/sql/`**; старый код не ломается, пока вы не переключаете запросы на новые поля.

| Этап | Таблица | Что происходит |
|------|---------|----------------|
| **Уже есть (init_db)** | **`knowledge_base`** | Базовые колонки: `id`, `ts`, `ticker`, `source`, `content`, `sentiment_score`, `insight`, `event_type`, `importance`, `link`, `ingested_at`, опционально **`embedding vector(768)`**, **`outcome_json`**. |
| **Уже есть (init_db)** | **`quotes`** | По смыслу схемы LSE — **дневные** OHLCV (`date`, `ticker`, open/high/low/close, volume) + **техиндикаторы** (RSI, MACD, Bollinger, ADX, Stochastic и др.), уникальность **`(date, ticker)`**. Типичное заполнение — **одна строка на торговый день на тикер**. **Игра 5m** использует **пятиминутные** свечи из **API в рантайме** (yfinance и т.п.), а не полный архив 5m-баров в `quotes`. |
| **010** | **`knowledge_base`** | **Добавляются** (если ещё нет): `exchange`, `symbol`, `external_id`, `content_sha256`, `raw_payload` (JSONB); частичные **UNIQUE** на `external_id` и `(ticker, link)` при непустых значениях; бэкфилл `symbol = ticker`. |
| **020** | **`market_bars_daily`**, **`market_bars_1h`** | **Новые** таблицы «сырых» OHLCV с **`exchange` + `symbol`** (см. §6.2): **параллельно** `quotes`. **`market_bars_daily`** — для дневных forward log-return и бенча; **`market_bars_1h`** — **для новостного контура** (см. §6.2), если нужна внутридневная привязка без хранения полного **5m** архива в PG. |
| **030** | **`news_signal_log`** | **Новая** таблица: снимок решения (`decision_ts_utc`, `symbol`, `model_id`, `bias`, `confidence`, **`knowledge_base_ids[]`**, опц. `tech_bias`, `trade_id`, `extra`). |
| **040** | **`knowledge_base`** | **Индексы:** BRIN/`ticker+ts`, `exchange+ts`, `symbol+ts`, GIN по `raw_payload` — только ускорение выборок, схема строки та же. |
| **Дальше (по необходимости, отдельные миграции)** | **`knowledge_base`** | Поля вроде **`story_cluster_id`**, **`cluster_canonical`** (см. §8.6) — когда зафиксируете правила L2-кластеров. |
| **Дальше (опционально)** | **`news_story_cluster`** | Кластер как сущность: id, время, опц. центроид эмбеддинга — удобно пересчитывать офлайн. |
| **Дальше (опционально)** | **`news_document`** | Вынести длинный body из горячей строки KB; в **`knowledge_base`** — ссылка + excerpt (§6.1). |

**Итог:** «основная» таблица новостей остаётся **`knowledge_base`**, она **расширяется колонками**; появляются **`market_bars_*`** и **`news_signal_log`**; кластеры и вынесенный текст — **следующий слой**. Таблица **`quotes`** остаётся рабочей для **дневной** техники и отчётов; **`market_bars_*`** её **не заменяют** на первом этапе — это отдельный слой под цены «как у биржи» и бенч.

### 6.1 Что поправить в заполнении новостей

| Проблема | Направление |
|----------|-------------|
| Дубликаты одной статьи | Вводить **`external_id`** (хэш url+title или id провайдера), `UNIQUE`; импорт — `INSERT … ON CONFLICT DO NOTHING/UPDATE`. |
| Длинный текст в горячей строке | Таблица **`news_document`** (id, body, hash) или объектное хранилище; в `knowledge_base` — **ссылка** + краткий excerpt. |
| Нет привязки к бирже | Поля **`exchange` / `asset_class`** (NYSE, LSE, COMEX…) и нормализованный **`symbol`** (как в ценах). |
| Ручной поток | Крон: **newsapi_fetcher** (или NYSE-пайплайн) → **staging** → валидация → merge в prod-таблицу; алерты при провале. |

Имеет смысл либо **расширить** `knowledge_base`, либо ввести **`news_event`** (каноническая сущность) и оставить KB как представление для бота — главное: один **источник правды** для tradenews-экспорта.

### 6.2 Свечи: `quotes`, игра **5m**, и `market_bars_*` под **новости**

- **`quotes`** — **дневной** слой + индикаторы (см. §6.0). Для **GAME_5m** это не замена минутным данным: решения по **5-минутным** свечам строятся из **живого** потока/API, а не из таблицы «все 5m-бары за год» в PG.

- **`market_bars_daily`** (миграция `020`) — **минимум для новостного бенча и tradenews:** дневные close → **forward log-return** 1d/3d/5d(6d) в одной семантике с `valuation.py`. Upsert из **yfinance / Polygon / IB**; ключ `(exchange, symbol, trade_date)`.

- **`market_bars_1h`** — **предлагается для новостного контура**, когда дневного close мало, а **класть в PG все 5m** избыточно (объём, крон, шум):  
  - привязка **`knowledge_base.ts`** / `decision_ts` к **ближайшему часовому** бару (UTC, как в `020`);  
  - внутридневная **оценка реакции** цены на новость без полного 5m-архива;  
  - подготовка к правилам «новость до/после открытия», лаг ленты.  
  Начинать с **тикеров GAME/портфеля** и только нужный горизонт дней назад.

- **Отдельная таблица `market_bars_5m` в PG** — **не обязательна** для старта: 5m остаётся в **игровом** пайплайне через API. Вводить хранение 5m в базе имеет смысл только при явном исследовании (например, офлайн-совмещение каждой новости с точным баром) — это уже отдельная миграция и нагрузка.

Переносить индикаторы из `quotes` в `market_bars_*` **не требуется**; дублировать **дневной** close в `market_bars_daily` для бенча — нормально.

### 6.3 Связка LSE ↔ NYSE (tradenews)

- **Один PostgreSQL** (lse) + NYSE-скрипты пишут в те же таблицы `news_*` / `bars_*` **или**  
- NYSE кладёт **JSONL/Parquet** в общий бакет, ночной job в LSE **заливает** в PG.  
Экспорт для бенча: SQL/view → **JSONL точек** (`ticker`, `decision_ts`, `articles_snapshot` или путь к фикстуре) — как уже делает tradenews.

**Готовые миграции PostgreSQL** (поля к `knowledge_base`, `market_bars_*`, `news_signal_log`, индексы; совместимо с `init_db.py` и pgvector): репозиторий **lse**, каталог **`db/knowledge_pg/`** (см. `README.md` там; копия может лежать рядом с чекаутом **nyse**).

### 6.4 Схема «обучения» (без иллюзий про один промпт)

Это **не** обязательно fine-tune LLM с первого дня.

1. **Сбор:** каждая точка решения — строка **`news_signal_log`**: `decision_ts`, `symbol`, `model_id`, `bias`, `confidence`, **`knowledge_base_ids[]`**, опционально `tech_bias`, `trade_id`.  
2. **Метка доходности:** офлайн-джоб читает **`market_bars_daily`** (и при необходимости **`market_bars_1h`**), считает **`forward_log_return_1d/3d/6d`** по тем же правилам, что tradenews `valuation.py` (одинаковая семантика «close» и календаря).  
3. **Калибровка:** sklearn / pandas: признаки = выход LLM + число новостей + час дня; цель = знак доходности или квантиль → **Platt / isotonic / логрег**; отчёт ROC / Brier / Spearman на holdout по времени.  
4. **Тонкий fine-tune (опционально):** когда накоплены **тысячи** строк с стабильной разметкой — LoRA на парах (текст статей → метка), отдельно от LSE-крона.  
5. **Валидация:** разрез по **времени** и по **тикеру**, не случайное перемешивание.

Так **база новостей + база свечей** становятся фундаментом не «памяти RAG», а **воспроизводимой разметки** для калибровки и, при росте N, для дообучения.

### 6.5 Что храним «серьёзно» для калибровки, кроме того, что уже пишет игра при входе

**Уже есть (GAME_5m, без новостного слоя):** при бумажном входе в **`trade_history`** пишется BUY со **`strategy_name = GAME_5M`**, ценой, `signal_type` и **`context_json`** — снимок **технического** контекста на момент решения (импульс 2ч, RSI 5m, волатильность и т.д., см. `record_entry` в `lse/services/game_5m.py`). На выходе — строки SELL и итог сделки. Этого **достаточно**, чтобы учить/смотреть **технический** контур и CatBoost по `context_json`, но **нет** стабильной ссылки на **какие новости** и **какой news_bias** считались в ту же секунду.

**Дополнительно под новости и калибровку (слой «над» игрой, не заменяя `trade_history`):**

| Данные | Где | Зачем |
|--------|-----|--------|
| Тексты и метаданные новостей | **`knowledge_base`** (+ позже кластеры §8.6) | Источник правды для ленты, дедуп, эмбеддинги; по `id` восстанавливается вход LLM. |
| Снимок **новостного** сигнала | **`news_signal_log`** | `decision_ts_utc`, `symbol`, `model_id`, `bias`, `confidence`, **`knowledge_base_ids[]`**, `prompt_version`, опц. **`tech_bias`**, опц. **`trade_id`** — связка «решение по новостям ↔ сделка». Пока игра без LLM-новостей — лог можно писать из **бота/аналитики** или пустой хвост; при fusion — заполнять **`trade_id`** из `trade_history` после входа. |
| Цены для меток | **`market_bars_daily`** (и при необходимости **`market_bars_1h`**) | Подсчёт **forward log-return** в той же семантике, что tradenews; без дублирования OHLC внутри `context_json`. |
| Офлайн-метки (опционально) | **view / материализованная таблица / Parquet** (не обязательно новая сущность в проде) | Строка «точка»: `decision_ts`, `symbol`, `forward_lr_1d`, … — чтобы не пересчитывать бары при каждом эксперименте. |
| Коэффициенты калибровки, версия модели | **файлы в репо / S3 / маленькая таблица `model_calibration`** (по желанию) | Версионирование Platt/isotonic/логрег; в рантайме только **применение**, не обучение. |

**Сюжет: когда писать `news_signal_log` — при попытке входа, а не при закрытии.**

- **Канонично:** строка пишется в момент **решения по новостям до/вокруг входа** — когда собран снимок KB, вызван LLM (или правила), получены `bias` / `confidence`. Поле **`decision_ts_utc`** = это время; оно должно быть **≤** времени фактического BUY и согласовано с окном статей в промпте. **`trade_id`** подставляют **сразу после** успешного `record_entry` (или оставляют `NULL`, если сигнал посчитали, но в **BUY** не пошли — тоже полезно для калибровки «модель сказала X, сделки нет»).
- **Почему не при закрытии:** метка forward log-return отсчитывается от **момента, когда трейдер/бот «видел» рынок и новости** перед входом; лог только на выходе смешивает **постфактум** ленту с целевой переменной и ломает сравнимость с бенчем tradenews. Закрытие уже отражено в **`trade_history`** (SELL, PnL).
- **Исключение (отдельная задача):** если позже введёте явный слой «**новости на выходе**» (форсировать SELL по ленте), это **вторая** запись или флаг `phase=exit` в `extra` JSONB — не подменяет основной ряд для калибровки **входа**.

**Как это стыкуется:** офлайн-джоб (Python на сервере или ноут) делает `JOIN`: **`news_signal_log`** → **`knowledge_base`** по id массива → **`market_bars_*`** по времени и символу → признаки и целевая доходность → отчёт и, при необходимости, новый JSON с весами. **Полный текст новостей в `context_json` сделки не кладём** — только ссылки на KB, чтобы не раздувать строку и не плодить расхождения с дедупом.

**Настройка системы в двух словах:** прод продолжает писать **технику** в `trade_history`; параллельно по мере внедрения — **`news_signal_log`** + наполненная **KB** + **бары**; калибровка и бенч **читают** это трое и **не требуют** менять ядро записи входа, пока вы явно не добавите запись `news_signal_log` в тот же cron, что открывает сделку.

## 7. Роли трёх репозиториев и интеграция в бота LSE

Согласованная картина (обновлять по мере эволюции):

| Репозиторий | Роль сейчас |
|-------------|-------------|
| **tradenews** | Лаборатория: **тесты гипотез**, сравнение моделей, метрики vs forward log-return, отчёты. Минимальная зависимость от прода; ключи и тяжёлые прогоны — ноут/рабочая станция. |

Реализация поэтапная: сначала **read-only** подсказки в UI бота, затем правила входа/фильтры, без подмены ценового риск-контура.

## 8. Два контура новостей (LSE и NYSE) → одна каноническая KB в LSE без поломки игры

**Цель:** улучшать **качество импульса за счёт архитектуры данных** (дедуп, идентификаторы, бары, лог сигналов), а не «перепрошивать» GAME_5M в первый день. Игра **как сейчас** остаётся на технике; новости и LLM **не обязаны** попадать в условия входа, пока не включите отдельные флаги (например `GAME_5M_USE_LLM_NEWS` — по умолчанию выкл.).

### 8.1 Канон хранения

- **Один источник правды в проде:** PostgreSQL **LSE**, таблица **`knowledge_base`** + миграции из **`lse/db/knowledge_pg/`** (`010` — поля биржи/символа/внешнего id/дедуп, `020` — `market_bars_*`, `030` — `news_signal_log`). Подробности и порядок применения: [README в knowledge_pg](https://github.com/kobyzev-yuri/lse/blob/main/db/knowledge_pg/README.md).
- **NYSE-репозиторий** — полезен как **лаборатория** (промпты L0–L5, другой набор скриптов, локальный кэш). «Лучшее из двух миров» = **не две разные схемы в проде**, а **одинаковый контракт строки KB** + при необходимости **импорт** из NYSE-пайплайна в ту же БД (общий SQL-слой уже задуман как общий для LSE/NYSE).

### 8.2 Что переделать в LSE в первую очередь (минимальный риск для игры)

| Шаг | Действие | Почему не ломает игру |
|-----|----------|------------------------|
| **1** | Применить миграции `010`–`040` на сервере; почистить дубли под `(ticker, link)` / `external_id`, если миграция упадёт. | Только схема и индексы; код игры не читает новые колонки, пока вы сами не начнёте. |
| **2** | В **существующих** импортёрах (`newsapi_fetcher`, Investing, Alpha Vantage, `vector_kb`, `llm_news_fetcher`, бот) постепенно заполнять **`exchange`**, **`symbol`**, **`external_id`** (или стабильный хэш), **`raw_payload`** при наличии JSON от API. INSERT остаётся в `knowledge_base` — без новой таблицы на первом круге. | Старые клиенты читают те же `ts, ticker, content, …`; новые поля — опциональны. |
| **3** | Завести **`market_bars_daily`** (миграция `020`) и ночной/кроновый upsert из одного провайдера — **параллельно** legacy `quotes`, без смены логики 5m. | Forward log-return для бенча/калибровки и tradenews; не подмена intraday игры. |
| **4** | Писать **`news_signal_log`** только там, где уже есть экспериментальный вызов LLM по новостям (бот, аналитика, отдельный job) — не из крона входа 5m. | Игра не зависит от этой таблицы. |
| **5** | Когда снимок KB стабилен — **read-only** подсказки в Telegram / дашборде; затем §2 **fusion** и только потом опционально фильтр входа 5m. | Явное включение флагов; тех. путь без новостей сохраняется. |

### 8.3 Как стыковать «NYSE-новости» с LSE

Практичные варианты:

1. **JSONL → KB (рекомендуемый первый этап):** пайплайн NYSE **не трогаем** в глубину — он пишет **JSONL** (или stdout в файл) → отдельный **импортёр** читает построчно и делает **upsert в `knowledge_base`** (`ON CONFLICT` по `external_id` / `(ticker, link)`). **Плюсы:** изоляция сбоев (сломался PG — файл всё равно есть), проще отладка и повторный прогон того же файла, **одно место** для маппинга NYSE→KB; **минусы:** второй шаг в cron, диск. После того как схема полей и дедуп стабильны, при желании можно сжать в один процесс (п. 2).  
2. **Прямая запись в PG из NYSE:** после сбора сразу вызывать «статья → upsert `knowledge_base`» при наличии `DATABASE_URL`. Имеет смысл **после** отладки маппинга на п. 1, чтобы не смешивать отладку пайплайна и отладку SQL.  
3. **Пока не сливать потоки:** улучшить только LSE-импорт; NYSE оставить для офлайн-бенча и переноса **правил отбора/дедупа** в код LSE.

Формат полей NYSE vs то, что ждёт tradenews/LSE, может отличаться — см. заметку в **lse** [`docs/EXPORT_DATASET_POSTGRES_TRADENEWS.md`](https://github.com/kobyzev-yuri/lse/blob/main/docs/EXPORT_DATASET_POSTGRES_TRADENEWS.md) (нужен **конвертер** между схемами).

### 8.4 Явно не трогаем на первом этапе

- Ядро **`get_decision_5m` / recommend_5m** и правила **BUY/STRONG_BUY** по технике.  
- **`GAME_5M_TAKE_MOMENTUM_FACTOR`** и прочие **ценовые** параметры импульса 2ч — не смешивать с новостным слоем (см. §1).  
- Включение **`GAME_5M_USE_LLM_NEWS`** — только после готовности снимка KB + хотя бы ручной проверки.

### 8.5 Обоснование: зачем PostgreSQL, если в NYSE отказались от БД ради «только актуальных» новостей

Это **не противоречие**, пока разделить два разных вопроса:

| Вопрос | Суть |
|--------|------|
| **Что попадает в сигнал?** | Только новости **в допустимом окне до `decision_ts`** — без подмешивания устаревшего шума в промпт. Это **политика снимка** (фильтр по времени, топ-K, дедуп внутри окна). |
| **Где хранить факт забора и разметки?** | Нужен **воспроизводимый след**: что именно считалось «актуальным» в момент решения, откуда пришло, как сшито с барами. Это **инфраструктура честности** метрик и продакшена. |

**Отказ от БД в NYSE** обычно означает: лабораторный контур «запустил скрипт — получил ответ», один потребитель, минимум зависимостей. Это разумно для **экспериментов**. В **LSE** уже есть бот, кроны, дашборд, лимиты API и необходимость **сравнивать** прогоны бенча и калибровки во времени — без общего хранилища это превращается в несопоставимые логи и дубли между источниками.

**Что даёт PostgreSQL (и почему это не «усложнение ради усложнения»):**

1. **Воспроизводимость** — для той же точки `(symbol, decision_ts)` можно восстановить **список id** статей и содержание, попавшее в модель; иначе Spearman и A/B по промптам **несостоятельны** («мы что-то поменяли, а вход больше не повторить»).
2. **Один ingest — несколько потребителей** — не дергать NewsAPI/Investing отдельно для бота, крона и офлайн-экспорта; экономия лимитов и единое дедуплицированное тело текста.
3. **Дедуп и идемпотентность** — одна статья из разных каналов не должна трижды искажать sentiment и метрики; уникальные индексы и `ON CONFLICT` проще, чем ручная уборка файлов.
4. **Стыковка с рынком** — те же `symbol` / `exchange`, джойн к **`market_bars_*`** для **forward log-return** (в лог-доходностях, с учётом издержек в проде). БД здесь **не предлагает торговать по архиву** — она даёт **разметку** для оценки «чистоты» сигнала.
5. **Аудит и доверие** — при спорном входе можно показать: вот `knowledge_base_ids`, вот версия промпта (`news_signal_log`), вот время; без этого обвинения в «чёрном ящике» обоснованы.

**Когда можно обойтись без PG:** изолированные офлайн-прогоны, один автор, нет общей прод-БД с LSE — тогда файлы/кэш допустимы (как в NYSE-лабе).

**Итог для обсуждений:** PostgreSQL — это не отказ от принципа «сигнал только из свежего окна», а **способ гарантировать, что «свежее» однозначно зафиксировано, не продублировано и сопоставимо с ценой**. Усложнение оплачивается **сравнимостью экспериментов, экономией API и готовностью к импульсу в проде** без магии в разрозненных файлах.

### 8.6 Сбор как в NYSE → KB, дедуп, векторы и **кластеры** (дубликаты — частный случай)

**Целевой контур:** те же идеи, что в NYSE-скриптах (агрегация лент, фильтры, гео/макро при необходимости), но **запись в LSE `knowledge_base`** + последующая обработка, а не «только в памяти». **Векторизация** (`embedding` в KB или отдельная таблица чанков позже) нужна и для поиска по смыслу, и как основа **семантической близости** между статьями.

**Уровни «схлопывания» шума** (внедрять поэтапно):

| Уровень | Что ловит | Типичные средства |
|---------|-----------|-------------------|
| **L0 — идентичность** | Та же статья (тот же URL, тот же `external_id`, тот же нормализованный текст) | Уникальные индексы `010`, `content_sha256`, `(ticker, link)` |
| **L1 — почти дубликат** | Тот же пресс-релиз, слегка другой заголовок / UTM в URL | Нормализация URL, simhash заголовка+лида, порог по расстоянию |
| **L2 — одна «история»** | Разные формулировки и источники про **одно событие** (как в экспериментах NYSE по **геополитическим кластерам**) | Эмбеддинги + порог косинусной близости, или HDBSCAN/иерархия по времени+вектору, или правило «в окне ±N часов и similarity > τ» |

**Да — для L2 проще опираться на уже существующие эмбеддинги.** В LSE уже есть **`knowledge_base.embedding`** и **pgvector** (`services/vector_kb.py`): тот же вектор, что для RAG/поиска, можно использовать как **пространство близости** для кластеров — отдельная «модель только под кластеризацию» не обязательна. Типичный путь: для каждой новой строки после INSERT искать соседей по **косинусной близости** (и желательно **временному окну**), присваивать `story_cluster_id` или создавать новый кластер; центроид кластера — среднее эмбеддингов (опционально, для отчётов).

**Зачем тогда L0/L1, если есть векторы:** они **дешевле и детерминированнее** для **точных** дублей (тот же URL, тот же текст) — не тратим эмбеддинг на заведомый повтор и не полагаемся на случайные ошибки модели (редко: близкие векторы у разных тем или далёкие у пересказа одной истории). Практичная схема: **сначала L0/L1**, потом эмбеддинг **для канонической строки** и L2-соседи.

**Дубликаты** — это **нижний подуровень L0–L1**. **Кластер** шире: «все статьи про одну цепочку санкций / один инцидент», чтобы в снимок для LLM не ушло **пять пересказов одного факта** и не завысился «вес» темы.

**Как отразить в данных (концепция):**

- У каждой строки KB — опционально **`story_cluster_id`** (UUID или bigserial), общий для всех членов кластера; одна строка помечается как **`cluster_canonical`** (representative) для показа/промпта.  
- Либо отдельная таблица **`news_story_cluster`** (`id`, `created_at`, `label`, `centroid_embedding` опционально) и FK с KB — удобнее для пересчёта кластеров офлайн.  
- **Политика снимка:** в промпт отдаём **не все** члены кластера, а например **канон + 1 контрастный источник** или только канон — правило фиксируем в коде/конфиге, чтобы метрики были воспроизводимы.

**Связь с геополитикой NYSE:** там кластеризация уже мыслилась как способ **не считать десять заголовков одним и тем же риском**. Тот же приём переносится на **любые** пачки новостей (earnings, регуляторика, войны/энергия): дедуп по строке ≠ учёт **семантической** избыточности.

**Порядок работ:** сначала стабильный **L0** в проде → затем **эмбеддинги** для существующих строк → эксперименты **L2** офлайн (подбор τ и окна) → только потом автоматическое присвоение `story_cluster_id` в горячем ingest. Игра и техника по-прежнему не зависят от кластеров, пока импульс не читает их в входе.

### 8.7 Деплой NYSE-скриптов новостей на сервер и вывод LSE-кронов

**Да, разумная схема:** на той же машине (или рядом в сети с PostgreSQL), **отдельный каталог** с **`git clone`** репозитория **nyse** с GitHub — например `/opt/nyse` или `~/nyse`, **не обязательно** внутри дерева **lse**. Так проще **независимые** `git pull`, свои зависимости (`venv`) и точка входа для новостей, не трогая образ/релиз LSE при каждой правке пайплайна.

**Важно:** отдельный clone **сам по себе не решает PG**. На старте логичнее **§8.3 п. 1**: cron **(а)** `nyse … > /var/lib/nyse/out/run_*.jsonl`, **(б)** `python import_nyse_jsonl_to_kb.py …` с `DATABASE_URL` (скрипт в **nyse** или **lse** — один контракт колонок). **Встроенный sink в процесс NYSE** (прямой upsert без файла) — **второй этап**, когда маппинг и конфликты уже отлажены на JSONL.

Переходный режим: сохранять **JSONL** (аудит/реплей) **и** заливать в PG; при сбое импорта можно **переиграть** тот же файл без повторного дергания API.

| Тема | Рекомендация |
|------|----------------|
| **БД** | Тот же **`DATABASE_URL`**, что у LSE (или сеть до того же Postgres). Запись только в **`knowledge_base`** по контракту миграций `010+` (`external_id`, `exchange`, `symbol`, …). |
| **Секреты** | Файл **`nyse/config.env`** (или `.env`) вне git; при необходимости дублировать только ключи источников; общий PG — один раз. |
| **Запуск** | **systemd timer**, **cron** пользователя `deploy` или отдельный контейнер «только cron+python» с volume на clone nyse. Логи в `nyse/logs` или journald. |
| **LSE не ломаем** | Контейнер/процесс **lse** не обязан знать про путь к nyse; единственный контракт — **PG**. |

**Постепенный отказ от старых скриптов в lse** (newsapi_fetcher, investing, дубли кронов):

1. **Пересечение:** какое-то время держать **оба** контура, опираясь на **L0-дедуп** в KB (`external_id`, `(ticker, link)`), чтобы не раздувать таблицу и не портить метрики.  
2. **По одному источнику:** переключить источник X на nyse → **отключить** соответствующую строку в **crontab LSE** / `fetch_news_cron` для X.  
3. **Пустой хвост:** когда все нужные источники в nyse — в LSE оставить только то, что **не новости** (или тонкий вызов, если останется один legacy-путь). Код в `services/` не удалять в один день: пометить **deprecated**, убрать из расписания, потом вычистить PR-ом.

**Не обязательно** тащить nyse как **submodule** внутрь lse на сервере: для прода удобнее **два соседних clone** и явные пути в cron. Submodule имеет смысл на **рабочей станции**, если хотите один `git status`.

Итог по §8: **главный рычаг** — привести **LSE `knowledge_base` + бары + лог** к воспроизводимому контракту; **NYSE** — источник идей и, при желании, второй ingest в **ту же** схему. Игра без новостей продолжает работать как сейчас.
