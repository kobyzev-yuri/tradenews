# План: новостной импульс — внедрение и проверки

Документ фиксирует **намеченный порядок работ** и **типы тестов** с учётом уже пройденного бенча (мало точек, разные горизонты, Gemini/Qwen сильнее llama 3b на game-срезе, ProxyAPI, отдельность ценового импульса в GAME_5M). План пересматриваем после каждого заметного прогона или смены промпта/моделей.

---

## 1. Факты из опыта (не забывать)

- **Операционная реальность LSE (2026):** **`knowledge_base` новостей** в основном накапливается импортом/ботом, но **автоматическая игра сейчас на технике**; новости + LLM использовались **точечно человеком** (запросы в аналитику), часто как **ограничитель агрессии** (BUY / STRONG_BUY после разбора ленты). Значит схему KB и пайплайн наполнения **можно править под будущий сигнал** без страха «сломать уже работающий новостной автотрейд» — его как раз предстоит **включать осознанно** (§2, §7).
- **Корреляции:** в решениях LLM уже смешиваются **новости и техника** и учитывается **корреляция цен по тикерам** (портфель и контекст в промпте). Отдельно **не делалась** модель совместного появления/тематики новостей по тикерам («новостная корреляция») — при расширении KB и fusion это отдельный предмет гипотез.
- Оценка — **не обучение**: метрики по **тому JSONL точек**, который собрали; **тикерный состав** задаётся файлом датасета, а не отдельным «списком обучения» в коде бенча.
- **Малое N** (десятки точек) → смотреть **CI / p-value**, повторять на другом периоде; «победитель модели» может поменяться.
- **Новости ≠ вся цена**: техника и режим рынка могут перебивать новостной слой — в проде нужен **явный слой смешивания**, а не замена техники.
- **`GAME_5M_TAKE_MOMENTUM_FACTOR`** — про **ценовой** импульс 2ч; новости подключать **отдельными** правилами (фильтр, размер, опционально цель выхода), не смешивать смысл с этим коэффициентом.
- **Объём и качество потока новостей** — отдельное узкое место: мало статей, дубликаты, лимит токенов, задержка данных.

## 2. Внедрение (по этапам)

| Этап | Действие | Критерий готовности |
|------|----------|---------------------|
| **A** | При «решении» сохранять снимок сигнала: `ticker`, `decision_ts_utc`, `news_bias`, `confidence`, `model_id`, ссылки/id статей (полный текст — по возможности вне горячей БД). | Можно воспроизвести строку уровня EvaluationRow офлайн. |
| **B** | Слой **fusion** с техникой: использовать поле `tech_bias` в датасете / то же в проде; правило согласия знаков или взвешенный скор. | Снижение конфликтных входов без ухудшения метрик на holdout-срезе. |
| **C** | Песочница **GAME_5M**: новости → фильтр входа и/или размер; изменение тейка — только после A/B, отдельный флаг/коэффициент. | Нет подмены `momentum_2h` новостным полем. |
| **D** | Регулярный мониторинг: сводка vs факт (хотя бы hit по знаку + разрез по тикерам); сравнение **Qwen vs Gemini** на новых точках. | Регресс сигнала виден за 1–2 недели. |

## 3. Дополнительные тесты (узкие места по новостям)

Все пункты — **офлайн** на том же или расширенном JSONL, без обязательной смены модели.

1. **Страты по числу статей** на точку (например 0–2 / 3–7 / 8+): Spearman и hit отдельно.  
2. **Усечение контекста:** заголовки only / топ-K по времени vs полный текст — проверка упора в **токены** и шум.  
3. **Дедуп** URL/заголовка vs сырой поток — влияние **повторов** на bias.  
4. **Лаг данных:** сдвиг `decision_ts` (симуляция поздней ленты) — устойчивость.  
5. **Новый временной срез** при тех же тикерах — не переобучиться на одном календарном окне.  
6. **Порог \|bias\|** (0.03 / 0.05 / 0.1): баланс покрытие vs hit rate.

## 6. Данные в LSE / NYSE: новости, свечи, схема «обучения»

Сейчас в LSE новости часто попадают в **`knowledge_base`** (`ts`, `ticker`, `source`, `content`, `sentiment_score`, `insight`) через `news_importer.py` и бота. Для **точного** импульса и последующей калибровки этого мало: нужны **дедуп**, **внешний id/url**, **часовой пояс**, **сырые свечи** с идемпотентным добором.

### 6.1 Что поправить в заполнении новостей

| Проблема | Направление |
|----------|-------------|
| Дубликаты одной статьи | Вводить **`external_id`** (хэш url+title или id провайдера), `UNIQUE`; импорт — `INSERT … ON CONFLICT DO NOTHING/UPDATE`. |
| Длинный текст в горячей строке | Таблица **`news_document`** (id, body, hash) или объектное хранилище; в `knowledge_base` — **ссылка** + краткий excerpt. |
| Нет привязки к бирже | Поля **`exchange` / `asset_class`** (NYSE, LSE, COMEX…) и нормализованный **`symbol`** (как в ценах). |
| Ручной поток | Крон: **newsapi_fetcher** (или NYSE-пайплайн) → **staging** → валидация → merge в prod-таблицу; алерты при провале. |

Имеет смысл либо **расширить** `knowledge_base`, либо ввести **`news_event`** (каноническая сущность) и оставить KB как представление для бота — главное: один **источник правды** для tradenews-экспорта.

### 6.2 Свечи: дневные обязательно, часовые — по необходимости

**Минимум для лог-доходностей 1d/3d/5d(6d):** таблица **`bars_daily`**:  
`(symbol, exchange, trade_date, open, high, low, close, volume, source, ingested_at)` с **`PRIMARY KEY (symbol, exchange, trade_date)`** и upsert из **yfinance / Polygon / IB** (один выбранный провайдер на тип инструмента).

**Часовые (`bars_1h`)** — если нужны техника 5m-уровня **или** выход внутри дня в модели:  
`(symbol, exchange, bar_start_utc, ohlcv, source)` + то же **идемпотентное** обновление. Начинать можно **только с тикеров из GAME/портфеля**, не со всего рынка.

### 6.3 Связка LSE ↔ NYSE (tradenews)

- **Один PostgreSQL** (lse) + NYSE-скрипты пишут в те же таблицы `news_*` / `bars_*` **или**  
- NYSE кладёт **JSONL/Parquet** в общий бакет, ночной job в LSE **заливает** в PG.  
Экспорт для бенча: SQL/view → **JSONL точек** (`ticker`, `decision_ts`, `articles_snapshot` или путь к фикстуре) — как уже делает tradenews.

**Готовые миграции PostgreSQL** (поля к `knowledge_base`, `market_bars_*`, `news_signal_log`, индексы; совместимо с `init_db.py` и pgvector): репозиторий **lse**, каталог **`db/knowledge_pg/`** (см. `README.md` там; копия может лежать рядом с чекаутом **nyse**).

### 6.4 Схема «обучения» (без иллюзий про один промпт)

Это **не** обязательно fine-tune LLM с первого дня.

1. **Сбор:** каждая точка решения — строка **`signal_log`**: `decision_ts`, `symbol`, `model_id`, `bias`, `confidence`, `news_ids[]`, опционально `tech_bias`, `trade_id`.  
2. **Метка доходности:** офлайн-джоб читает `bars_daily` (и при необходимости `bars_1h`), считает **`forward_log_return_1d/3d/6d`** по тем же правилам, что tradenews `valuation.py` (одинаковая семантика «close» и календаря).  
3. **Калибровка:** sklearn / pandas: признаки = выход LLM + число новостей + час дня; цель = знак доходности или квантиль → **Platt / isotonic / логрег**; отчёт ROC / Brier / Spearman на holdout по времени.  
4. **Тонкий fine-tune (опционально):** когда накоплены **тысячи** строк с стабильной разметкой — LoRA на парах (текст статей → метка), отдельно от LSE-крона.  
5. **Валидация:** разрез по **времени** и по **тикеру**, не случайное перемешивание.

Так **база новостей + база свечей** становятся фундаментом не «памяти RAG», а **воспроизводимой разметки** для калибровки и, при росте N, для дообучения.

## 7. Роли трёх репозиториев и интеграция в бота LSE

Согласованная картина (обновлять по мере эволюции):

| Репозиторий | Роль сейчас |
|-------------|-------------|
| **tradenews** | Лаборатория: **тесты гипотез**, сравнение моделей, метрики vs forward log-return, отчёты. Минимальная зависимость от прода; ключи и тяжёлые прогоны — ноут/рабочая станция. |
| **lse (сервер)** | **Мягкая интеграция** с **`knowledge_base`** и кронами импорта: миграции `db/knowledge_pg/`, улучшение заполнения, опционально **`news_signal_log`**. Учитывать, что **игра по умолчанию технич.**, KB в горячем контуре входа **пока слабо задействована** — изменения в БД в первую очередь готовят **данные и воспроизводимость** для будущего импульса, а не переключают прод сразу. |
| **nyse** | **Демонстрация и отладка качества сигнала** на разных моделях (в т.ч. Ollama на ноуте): пайплайн L0–L5, промпты, локальный кэш; зеркало SQL-слоя для удобства чекаута. |

**Вектор дальше:** усилить **Telegram-бота lse**, который сейчас по сути **технический**, за счёт **новостного импульса**: подмешивать к карточкам/решениям агрегированный сигнал из KB + (опционально) тот же LLM-путь, что отработан в nyse/tradenews, с **fusion** относительно техники (согласие знаков, пороги, размер — см. §2). Реализация поэтапная: сначала **read-only** подсказки в UI бота, затем правила входа/фильтры, без подмены ценового риск-контура.
