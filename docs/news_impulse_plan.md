# План: новостной импульс — внедрение и проверки

Документ фиксирует **намеченный порядок работ** и **типы тестов** с учётом уже пройденного бенча (мало точек, разные горизонты, Gemini/Qwen сильнее llama 3b на game-срезе, ProxyAPI, отдельность ценового импульса в GAME_5M). План пересматриваем после каждого заметного прогона или смены промпта/моделей.

---

## 1. Факты из опыта (не забывать)

- Оценка — **не обучение**: метрики по **тому JSONL точек**, который собрали; **тикерный состав** задаётся файлом датасета, а не отдельным «списком обучения» в коде бенча.
- **Малое N** (десятки точек) → смотреть **CI / p-value**, повторять на другом периоде; «победитель модели» может поменяться.
- **Новости ≠ вся цена**: техника и режим рынка могут перебивать новостной слой — в проде нужен **явный слой смешивания**, а не замена техники.
- **`GAME_5M_TAKE_MOMENTUM_FACTOR`** — про **ценовой** импульс 2ч; новости подключать **отдельными** правилами (фильтр, размер, опционально цель выхода), не смешивать смысл с этим коэффициентом.
- **Объём и качество потока новостей** — отдельное узкое место: мало статей, дубликаты, лимит токенов, задержка данных.

---

## 2. Внедрение (по этапам)

| Этап | Действие | Критерий готовности |
|------|----------|---------------------|
| **A** | При «решении» сохранять снимок сигнала: `ticker`, `decision_ts_utc`, `news_bias`, `confidence`, `model_id`, ссылки/id статей (полный текст — по возможности вне горячей БД). | Можно воспроизвести строку уровня EvaluationRow офлайн. |
| **B** | Слой **fusion** с техникой: использовать поле `tech_bias` в датасете / то же в проде; правило согласия знаков или взвешенный скор. | Снижение конфликтных входов без ухудшения метрик на holdout-срезе. |
| **C** | Песочница **GAME_5M**: новости → фильтр входа и/или размер; изменение тейка — только после A/B, отдельный флаг/коэффициент. | Нет подмены `momentum_2h` новостным полем. |
| **D** | Регулярный мониторинг: сводка vs факт (хотя бы hit по знаку + разрез по тикерам); сравнение **Qwen vs Gemini** на новых точках. | Регресс сигнала виден за 1–2 недели. |

## 3. Дополнительные тесты (узкие места по новостям)

Все пункты — **офлайн** на том же или расширенном JSONL, без обязательной смены модели.

1. **Страты по числу статей** на точку (например 0–2 / 3–7 / 8+): Spearman и hit отдельно.  
2. **Усечение контекста:** заголовки only / топ-K по времени vs полный текст — проверка упора в **токены** и шум.  
3. **Дедуп** URL/заголовка vs сырой поток — влияние **повторов** на bias.  
4. **Лаг данных:** сдвиг `decision_ts` (симуляция поздней ленты) — устойчивость.  
5. **Новый временной срез** при тех же тикерах — не переобучиться на одном календарном окне.  
6. **Порог \|bias\|** (0.03 / 0.05 / 0.1): баланс покрытие vs hit rate.

## 4. Бенчмарк: пара Gemini + Qwen (в т.ч. 14B)

После `ollama pull qwen2.5:14b`:

```bash
# вариант через env:
# TRADENEWS_EVAL_MODEL_SPECS=qwen2.5:14b google:gemini-2.0-flash

PYTHONPATH=. python scripts/run_model_benchmark.py \
  --build datasets/points/<points>.jsonl \
  --out-jsonl runs/eval_gemini_qwen14.jsonl \
  --report-json runs/benchmark_gemini_qwen14.json \
  --models qwen2.5:14b google:gemini-2.0-flash
```

Итог зафиксировать краткой сводкой в **`docs/reports/`** (только метрики и выводы, без JSONL), по образцу существующих отчётов.
